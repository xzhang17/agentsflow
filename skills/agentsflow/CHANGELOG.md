# Agents Flow Skill — Changelog

- v3.1.0 (2026-07-20): dropped backward-compatibility machinery and made the skill forward-only. Collapsed the four version stamps (skill + three schema numbers) into a single skill-version stamp used for provenance only; removed the `Workflow schema`/`Profile schema`/`Execution-mode schema` declarations, the template slots for them, and the immutability/legacy-schema-regeneration language, since workflows are now authored fresh every run and no longer negotiate compatibility with previously generated files. Folded the standalone `references/latex-cleanup.md` into a **Post-success cleanup** subsection of the `artifact-document-latex` profile and deleted the file. Optimized the cleanup for completeness: `latexmk -c` is now followed by a bounded known-extension sweep across the whole resolved build boundary (root plus every `\input`/`\include`d-source and output/aux directory, recursively), removing `.bbl`, per-chapter `.aux`, and biber/glossary/index/beamer intermediates that the previous `.fls`-inventory-only approach left behind, while never touching `.pdf`, sources, figures, or designated deliverables. Updated the skill, safety, templates, workflow-authoring, execution-modes, and profile pointers accordingly.

- v3.0.7 (2026-07-20): removed the legacy `references/modes.md` tombstone per user decision; its content had been split into `profiles.md` and `execution-modes.md` at v3.0.0 and no current-contract run loads it. A frozen v2-era workflow that still names `references/modes.md` now fails on a missing file and must be regenerated; the compatibility sentences in `SKILL.md`, `execution-modes.md`, and `workflow-authoring.md` were reworded to state this. Workflow, profile, and execution-mode schemas remain unchanged.

- v3.0.6 (2026-07-20): repaired remaining one-reply-rule contradictions and review-loop gaps found in a full cross-file audit. A questionnaire reply that materially expands scope now deterministically blocks instead of referencing an unreachable decision path; the scripted-review `requires-weakening` gap uses the remaining decision request only when the single user-decision reply is unspent and otherwise blocks. Capped ADVISOR-light at the same two-round maximum as scripted review, generalized the corresponding blocked-run trigger, and added a review-round field to the batch structural-review record. Added an explicit stale-tuple rule for `batch-anchored` items whose files earlier checklist items edit. Consolidated the divergent Decision record shapes into the single canonical block in `templates.md`, aligned the VISION validation unit to assigned pairs/standalone sites, clarified when VISION remains optional, documented the late-decision waiting status, and fixed minor wording drift (`after rendering`, templates procedure pointers). Workflow, profile, and execution-mode schemas remain unchanged.

- v3.0.5 (2026-07-17): resolved post-v3.0.4 VISION and validation contract ambiguities. PLAN now assigns unique evidence identifiers and explicit comparison pairs or standalone sites, while VISION may choose bounded normalized crop coordinates only within crop-enabled assigned pages and criteria. VISION reports are keyed to those identifiers, uncorrectable required `VISION_FAIL` outcomes deterministically block instead of floating between failed and blocked states, and successful image discrepancies remain validation failures. Made LaTeX page inspection conditional on a visual requested result, centralized blocked-run triggers in `safety.md`, and aligned the skill, profiles, templates, launcher, PLAN, and VISION contracts. Workflow, profile, and execution-mode schemas remain unchanged.

- v3.0.4 (2026-07-17): made required PDF/page fidelity self-contained in VISION without broadening its project authority. Updated PLAN, VISION, the PDF evidence profile, safety rules, launcher and report templates, agent definitions, and persistent role rules so PLAN fixes the evidence set while VISION renders and judges it; a failed required VISION review now remains failed/blocked with the exact error instead of falling back to PLAN image inspection, OCR, or an inferred provider/credit explanation. Workflow, profile, and execution-mode schemas remain unchanged.
